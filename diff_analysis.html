vllm-ascend  vs  vllm-ascend-lby

在 scripts/35B_mxfp8.sh 路径下的代码差异分析

模型：Qwen3.5-35B-A3B-W8A8-MXFP8-FULL-QUANT 投机解码：qwen3_5_mtp (num=2) 编译：FULL_DECODE_ONLY 并行：DP=1 / TP=1
主线 HEAD：c9966f12 解决精度问题 lby HEAD：045aeac1 fix2

0. TL;DR

  • lby 是性能优先的活跃开发分支，带 Qwen3.5 三合一融合 kernel、MXFP8 padding fix、ENPU 路径、hamming sparse、extract_hidden_states、ngram_npu 等新特性。
  • 主线 (vllm-ascend/) 是精度修复分支，commit 解决精度问题 显式 把那个融合 kernel 注释掉了，回退到分离 op 实现。
  • 同一份 35B_mxfp8.sh 跑出来：lby 通常更快，主仓数值更稳。
  • vllm-ascend-lby/ 根目录的 1、2、1.txt、2.txt、3.txt 都是 pip 安装日志，与代码差异无关。

1. Qwen3.5 Attention 前向 影响最大

文件：vllm_ascend/patch/worker/patch_qwen3_5.py  —  类 AscendQwen3NextAttention.forward

vllm-ascend (主线) 精度优先

  • 融合 Triton kernel triton_split_qkv_rmsnorm_mrope 整段被注释掉。
  • QKV split、q_norm/k_norm、rotary_emb 各自独立 op。
  • 每层多发 3–5 个 op，但数值稳定。

vllm-ascend-lby 性能优先

  • 对 model_type 含 qwen3_5 的模型走融合 kernel：split + (1 + norm) + MROPE 一把搞定。
  • 每层只 launch 一个 kernel，attention 入口前预处理大幅缩短。
直接影响：35B-A3B 这种每层都进 attention 的稠密结构，单 step launch 数差异显著；这是两个仓 throughput 差距的主要来源。

2. MXFP8 量化方法 数值 / 兼容

文件：vllm_ascend/quantization/methods/w8a8_mxfp8.py

变化点主线lby
weight_scale 形状 input_size // group_size cdiv(input_size, group_size)，且不能被 2 整除时 F.pad(..., (0,1))
MoE apply 接口 global_num_experts - global_redundant_expert_num get_moe_num_logical_experts(layer, ..., n_shared_experts)，
显式处理共享专家 + 冗余专家
fused_experts 数据类型 仅识别 bf16 兼容 torch.float8_e4m3fn（MXFP8 直传）
对应 commit：lby 的 d857777d [Ascend950][BugFix] Fix w8a8_mxfp8 weight_scale group count: use cdiv for non-divisible input_size。 主线在 input_size 不能被 group_size 整除时会少加载一列 scale，权重维度对不上就直接报错。

3. ACL Graph  FULL_DECODE_ONLY

文件：vllm_ascend/compilation/acl_graph.py  —  ACLGraphWrapper

lby 的 wrapper 多了两个构造参数：

def __init__(self, ..., *, use_eagle: bool = False, enable_enpu: bool = False): ...

把"replay 之前是否要 current_stream().synchronize()"的判定从 vllm_config 推导改为显式注入，并新增 ENPU 路径。lby 的 model_runner 里会读 C 侧环境变量：

_enpu = get_c_env("ENPU_ENABLE")
self.enable_enpu = _enpu is not None and _enpu.lower() == "true"

主线无 ENPU 路径，replay 同步仅依据 vllm_config.speculative_config.use_eagle()。

5. CPU Binding  enable_cpu_binding:true

  • 大部分是 f-string → lazy logging 的清理。
  • 唯一行为差异：lby 在分配 CPU 池时扫描 all_logic_npus，把"虽然不可见但其 NUMA 亲和 CPU 与本进程 cpuset 重叠"的 NPU 也参与冲突计算，最后再用 running_npu_list 过滤；主线只看可见 NPU。

一句话结论

lby 是性能优先的活跃开发分支（带 Qwen3.5 三合一融合 kernel、MXFP8 padding fix、ENPU/Hamming sparse/extract_hidden_states 等新特性）； 主仓这条 qwen3.5 a5 adapt → baseline → 解决精度问题 分支为了修精度显式禁掉了那个融合 kernel。

所以同一份 35B_mxfp8.sh 脚本在两个仓上跑出来的 吞吐和数值都会不一样  —  通常 lby 更快、主仓数值更稳。

lby 仓里的 1 / 2 / *.txt 看起来是 pip install 输出日志（python 3.11 + torch 2.10.0 + transformers 5.5.3 环境，路径 /home/l00939323/0519main/vllm-ascend），跟代码无关。

分析基于：diff -rq vllm-ascend vllm-ascend-lby 全量目录对比，并对脚本实际触发路径（MXFP8 量化、Qwen3.5 patch、ACL graph、cpu_binding、spec_decode、ascend_config、csrc）做了逐文件 diff。报告时间：2026-05-19。
